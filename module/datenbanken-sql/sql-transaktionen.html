Transaktionen Randwissen AP2

Einstieg

TowerTech GmbH verbucht eine Bestellung: Zuerst wird der Lagerbestand des Artikels reduziert, dann eine Bestellposition angelegt. Was passiert, wenn der Server genau zwischen diesen beiden Schritten abstürzt? Der Lagerbestand ist bereits verringert – aber die Bestellposition existiert nie. Die Datenbank widerspricht sich selbst: Ware ist „weg", aber niemand hat sie bestellt. Eine Transaktion verhindert genau das: mehrere SQL-Befehle werden als eine einzige, unteilbare Einheit behandelt – entweder laufen alle durch, oder keiner.

Konzept

Eine Transaktion fasst mehrere SQL-Befehle zusammen, die gemeinsam erfolgreich sein müssen. Sie ist die praktische Umsetzung des A in ACID (Atomarität): entweder alles oder nichts.

BefehlWirkung
BEGINstartet die Transaktion – alle folgenden Befehle gehören dazu (MySQL: auch START TRANSACTION)
COMMITspeichert alle Änderungen der Transaktion dauerhaft – danach gibt es kein Zurück mehr
ROLLBACKmacht alle Änderungen seit BEGIN rückgängig – als hätte die Transaktion nie stattgefunden

Ohne Transaktion – riskant, falls zwischen den beiden Befehlen ein Fehler passiert:

UPDATE lager SET bestand = bestand - 1 WHERE artikel_id = 1;
<< Server-Absturz hier — Bestand reduziert, aber keine Bestellposition! >>
INSERT INTO bestellposition (bestellung_id, artikel_id, menge) VALUES (500, 1, 1);

Mit Transaktion – entweder beide Befehle oder keiner:

BEGIN;
UPDATE lager SET bestand = bestand - 1 WHERE artikel_id = 1;
INSERT INTO bestellposition (bestellung_id, artikel_id, menge) VALUES (500, 1, 1);
COMMIT;

SAVEPOINT

Ein SAVEPOINT ist ein Zwischenpunkt innerhalb einer Transaktion. Man kann gezielt bis zu diesem Punkt zurückrollen, ohne die gesamte Transaktion abzubrechen:

BEGIN;
INSERT INTO bestellposition (bestellung_id, artikel_id, menge) VALUES (501, 2, 3);
SAVEPOINT nach_position;
UPDATE lager SET bestand = bestand - 999 WHERE artikel_id = 2; (Tippfehler bei der Menge)
ROLLBACK TO nach_position; (nur die fehlerhafte Buchung verwerfen)
COMMIT; (die INSERT-Bestellposition bleibt erhalten)

Autocommit

In den meisten DBMS läuft standardmäßig jeder einzelne SQL-Befehl automatisch in einer eigenen Mini-Transaktion (Autocommit = ein). Jedes INSERT, UPDATE, DELETE wird sofort dauerhaft gespeichert, sobald es ausgeführt wurde. Erst ein ausdrückliches BEGIN schaltet Autocommit für den folgenden Block ab und bündelt die Befehle zu einer echten Transaktion.

Praxisbeispiel

Eine Auszubildende bei TowerTech GmbH verbucht Lagerbewegungen im Editor oben:

  • Aufgabe 1 (COMMIT): Verbuche einen Verkauf eines Netzteils als Transaktion – Lagerbestand reduzieren und Bestellposition anlegen, dann COMMIT. Mit SELECT * FROM lager WHERE artikel_id = 1; prüfen: Der Bestand ist dauerhaft auf 11 gesunken.
  • Aufgabe 2 (ROLLBACK): Starte eine Transaktion, buche versehentlich 100 RAM-Riegel statt 1 ab (UPDATE lager SET bestand = bestand - 100 WHERE artikel_id = 2;), bemerke den Fehler und mache mit ROLLBACK alles rückgängig. Eine anschließende SELECT-Abfrage zeigt: der Bestand steht wieder bei 30, als wäre nichts passiert.
  • Aufgabe 3 (SAVEPOINT): Lege innerhalb einer Transaktion einen SAVEPOINT nach einer korrekten Bestellposition an, mache danach einen Fehler bei der Bestandsbuchung, rolle nur bis zum Savepoint zurück und committe den Rest.

Typische Prüfungsfrage: „Erklären Sie anhand eines Bank-/Lagerbeispiels, warum Transaktionen notwendig sind und welches ACID-Prinzip sie umsetzen." – Antwort: Transaktionen stellen sicher, dass zusammengehörige Änderungen entweder vollständig oder gar nicht ausgeführt werden – das ist Atomarität. Ohne Transaktion könnte ein Absturz zwischen zwei zusammengehörigen Befehlen einen inkonsistenten Datenbestand hinterlassen.

Merksatz

BEGIN startet, COMMIT speichert dauerhaft, ROLLBACK macht alles seit BEGIN rückgängig. SAVEPOINT erlaubt gezieltes Zurückrollen auf einen Zwischenpunkt, ohne die ganze Transaktion zu verwerfen. Ohne BEGIN läuft in den meisten DBMS jeder Befehl einzeln im Autocommit-Modus. Transaktionen sind die praktische Umsetzung der Atomarität aus ACID.
