ER-Modell & Kardinalitäten Kern AP2

Einstieg

Bevor ein Architekt ein Haus baut, zeichnet er einen Grundriss: wie viele Räume gibt es, wie sind sie miteinander verbunden, wo führt eine Tür von einem Zimmer ins nächste? Erst wenn der Plan stimmt, wird gemauert. Genauso wird eine Datenbank nicht direkt als Tabellen entworfen, sondern zuerst als Entity-Relationship-Modell (ER-Modell): ein Bauplan, der festhält, welche „Dinge" (Entitäten) es gibt und wie sie zueinander in Beziehung stehen – bevor daraus reale Tabellen mit Fremdschlüsseln werden.

Konzept

Ein ER-Modell besteht aus drei Grundbausteinen:

  • Entität – ein eigenständiges „Ding", über das Daten gespeichert werden (z. B. Kunde, Bestellung, Produkt). Wird zur Tabelle.
  • Attribut – eine Eigenschaft einer Entität (z. B. Kundenname, Bestelldatum). Wird zur Spalte.
  • Beziehung (Relationship) – wie zwei Entitäten miteinander verknüpft sind (z. B. „Kunde tätigt Bestellung").

Entscheidend ist dabei die Kardinalität einer Beziehung: wie viele Datensätze der einen Seite mit wie vielen Datensätzen der anderen Seite verknüpft sein können.

KardinalitätBedeutungBeispiel
1:1Ein Datensatz der einen Seite gehört zu genau einem der anderen SeitePerson – Personalausweis
1:nEin Datensatz der einen Seite kann zu mehreren der anderen Seite gehörenKunde – Bestellungen (ein Kunde, viele Bestellungen)
m:nDatensätze beider Seiten können mehrfach miteinander verknüpft seinBestellung – Produkt (eine Bestellung enthält mehrere Produkte, ein Produkt steckt in mehreren Bestellungen)

Bei der Umsetzung in Tabellen gilt eine feste Regel:

  • 1:n – der Fremdschlüssel steht auf der „n"-Seite (jede Bestellung bekommt eine Spalte Kunden_ID, die auf den Kunden verweist).
  • m:n – lässt sich nicht direkt über einen einzelnen Fremdschlüssel abbilden. Stattdessen wird eine Zwischentabelle (Verbindungstabelle, z. B. Bestellposition) eingefügt, die beide Fremdschlüssel enthält und die m:n-Beziehung in zwei 1:n-Beziehungen auflöst.

Damit Fremdschlüssel nicht ins Leere zeigen, gilt die referenzielle Integrität: ein Fremdschlüsselwert muss entweder auf einen tatsächlich existierenden Primärschlüssel in der referenzierten Tabelle zeigen oder NULL sein. Das DBMS lehnt Verstöße automatisch ab, z. B. beim Versuch, eine Bestellung mit einer nicht existierenden Kunden-ID einzufügen. Beim Löschen des referenzierten Datensatzes legt man über Löschregeln fest, was passieren soll: RESTRICT (Löschen verhindern, solange abhängige Zeilen existieren), CASCADE (abhängige Zeilen automatisch mitlöschen) oder SET NULL (Fremdschlüssel auf NULL setzen).

Praxisbeispiel

Ein IT-Systemhaus modelliert seine Warenwirtschaft: Kunden geben Bestellungen auf, eine Bestellung kann mehrere Produkte enthalten, und ein Produkt taucht in vielen Bestellungen auf.

  • Kunde – Bestellung (1:n): Die Tabelle Bestellung bekommt eine Spalte Kunden_ID als Fremdschlüssel, der auf Kunde.ID verweist.
  • Bestellung – Produkt (m:n): Da eine einzelne Spalte das nicht abbilden kann, entsteht die Zwischentabelle Bestellposition mit den Spalten Bestellung_ID, Produkt_ID und Menge. Sie löst die m:n-Beziehung in zwei 1:n-Beziehungen auf.
  • Ein Mitarbeiter versucht, eine Bestellposition mit einer Produkt_ID anzulegen, die es gar nicht gibt – das DBMS lehnt den Einfüge-Versuch ab, weil die referenzielle Integrität verletzt wäre.
  • Ein Kunde soll aus der Datenbank gelöscht werden, hat aber noch offene Bestellungen. Mit der Löschregel RESTRICT verweigert das DBMS das Löschen, solange Bestellungen existieren – so gehen keine Bestelldaten „verwaist" verloren.

Typische Prüfungsfrage: „Wie wird eine m:n-Beziehung in einem relationalen Datenbankmodell umgesetzt?" – Antwort: über eine Zwischentabelle mit zwei Fremdschlüsseln, die die m:n-Beziehung in zwei 1:n-Beziehungen auflöst.

Merksatz

Das ER-Modell ist der Bauplan vor der Umsetzung: Entitäten werden zu Tabellen, Attribute zu Spalten, Beziehungen zu Fremdschlüsseln. 1:n wird über einen Fremdschlüssel auf der „n"-Seite abgebildet, m:n über eine Zwischentabelle mit zwei Fremdschlüsseln. Referenzielle Integrität stellt sicher, dass Fremdschlüssel immer auf existierende Datensätze verweisen (oder NULL sind).
