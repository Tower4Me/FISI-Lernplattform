SQL & DML Kern AP2

Einstieg

Der Aktenschrank von TowerTech GmbH steht (siehe SQL & DDL) – Fächer für Abteilungen und Mitarbeitende sind angelegt. Jetzt geht es darum, tatsächlich Akten hineinzulegen, gezielt wieder herauszusuchen, falsche Einträge zu korrigieren und veraltete zu entsorgen. Genau das ist DML (Data Manipulation Language): nicht der Schrank selbst, sondern der tägliche Umgang mit dem, was drinsteht.

Konzept

DML umfasst die vier Befehle zum Lesen und Verändern von Daten: SELECT, INSERT, UPDATE und DELETE. Alle Beispiele beziehen sich auf die Tabellen abteilung und mitarbeiter aus der DDL-Einheit.

SELECT & WHERE

SELECT liest Daten, WHERE filtert nach einer Bedingung:

AusdruckBedeutung
SELECT * FROM mitarbeiter;alle Spalten, alle Zeilen
SELECT vollname, jahresgehalt FROM mitarbeiter;nur bestimmte Spalten
WHERE abteilung_id = 2nur Zeilen dieser Abteilung
WHERE jahresgehalt > 40000Vergleichsoperatoren
WHERE vollname LIKE 'T%'Namen, die mit „T“ beginnen
WHERE email IS NULLfehlende E-Mail-Adresse

ORDER BY & LIMIT

ORDER BY spalte DESC sortiert absteigend (Standard: ASC, aufsteigend). LIMIT n begrenzt die Ergebnisanzahl – z. B. „die drei bestbezahlten Mitarbeitenden":

SQL
SELECT vollname, jahresgehalt FROM mitarbeiter ORDER BY jahresgehalt DESC LIMIT 3;

INSERT – Daten einfügen

Neue Zeile mit ausdrücklich genannten Spalten (empfohlen):

INSERT INTO mitarbeiter (vollname, email, jahresgehalt, abteilung_id) VALUES ('Lea Fischer', 'lea.fischer@towertech.de', 41000, 1);

UPDATE – Daten ändern

UPDATE mitarbeiter SET jahresgehalt = 35500 WHERE vollname = 'Paul Nowak';

Achtung: UPDATE mitarbeiter SET jahresgehalt = 0; ohne WHERE setzt das Gehalt aller Mitarbeitenden auf 0. Vor jedem UPDATE/DELETE erst mit SELECT prüfen, welche Zeilen betroffen wären.

DELETE – Daten löschen

SELECT * FROM mitarbeiter WHERE vollname = 'Jonas Weber'; (erst prüfen)
DELETE FROM mitarbeiter WHERE vollname = 'Jonas Weber'; (dann löschen)

Aggregatfunktionen

FunktionBedeutung
COUNT(*)Anzahl aller Zeilen
SUM(jahresgehalt)Summe
AVG(jahresgehalt)Durchschnitt
MAX(jahresgehalt) / MIN(jahresgehalt)größter / kleinster Wert

GROUP BY & HAVING

GROUP BY fasst Zeilen zu Gruppen zusammen (z. B. je Abteilung), HAVING filtert diese Gruppen nach einer Aggregat-Bedingung:

SELECT abteilung_id, AVG(jahresgehalt) AS durchschnitt FROM mitarbeiter GROUP BY abteilung_id;
SELECT abteilung_id, AVG(jahresgehalt) AS durchschnitt FROM mitarbeiter GROUP BY abteilung_id HAVING AVG(jahresgehalt) > 40000;

WHERE filtert einzelne Zeilen vor der Gruppierung und kennt keine Aggregatfunktionen. HAVING filtert Gruppen nach GROUP BY und darf AVG(), COUNT() usw. verwenden.

Praxisbeispiel

Praxis „Azubi-Chaos": Die Personalliste von TowerTech GmbH hat sich geändert, und eine Auszubildende soll das im SQL-Editor oben korrigieren.

  • Aufgabe 1: Zeige alle Mitarbeitenden der Abteilung „Support" (abteilung_id 2), sortiert nach Gehalt absteigend.
  • Aufgabe 2: Eine neue Kollegin wurde vergessen: füge Lea Fischer ein (E-Mail lea.fischer@towertech.de, Gehalt 41000, Abteilung Netzwerktechnik).
  • Aufgabe 3: Paul Nowak hat eine Gehaltsverhandlung gewonnen: erhöhe sein Gehalt auf 35500.
  • Aufgabe 4: Jonas Weber hat gekündigt: erst mit SELECT prüfen, dann seinen Datensatz löschen.
  • Aufgabe 5: Berechne das Durchschnittsgehalt je Abteilung und zeige nur Abteilungen mit einem Durchschnitt über 40000 € (GROUP BY + HAVING).

Typische Prüfungsfrage: „Was ist der Unterschied zwischen WHERE und HAVING?" – Antwort: WHERE filtert einzelne Zeilen vor der Gruppierung und kennt keine Aggregatfunktionen; HAVING filtert Gruppen nach GROUP BY und darf Aggregatfunktionen wie AVG() verwenden.

Merksatz

DML liest und verändert Daten: SELECT (mit WHERE/ORDER BY/LIMIT) zum gezielten Lesen, INSERT zum Einfügen, UPDATE zum Ändern, DELETE zum Löschen – UPDATE/DELETE immer erst mit SELECT prüfen. Aggregatfunktionen (COUNT, SUM, AVG, MAX, MIN) fassen mehrere Zeilen zu einem Wert zusammen; GROUP BY bildet Gruppen, HAVING filtert diese Gruppen nach einer Aggregat-Bedingung.
