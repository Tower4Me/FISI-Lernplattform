Multitasking, Prozesse & Threads Kern AP1 AP2

Einstieg

Ein einzelner Koch in einer Küche kann eigentlich nur an einem Topf gleichzeitig rühren — wechselt er aber schnell genug zwischen mehreren Töpfen hin und her, wirkt es, als würde alles gleichzeitig kochen. Genau das macht eine CPU beim Multitasking. Beim kooperativen Modell gibt jeder Topf den Herd freiwillig ab, sobald er „fertig ist für den Moment“ — hängt ein Topf fest, blockiert er die ganze Küche. Beim präemptiven Modell gibt es einen strengen Küchenchef (den Scheduler), der jedem Topf nach einer festen Zeitspanne zwangsweise den Herd wieder wegnimmt — fair für alle, selbst wenn ein Topf nicht „freiwillig“ loslassen würde.

Konzept

Prozess, Task und Thread

BegriffBedeutung
Prozess ein laufendes Programm mit eigenem, vom Betriebssystem isoliertem Speicherbereich — ein Prozess kann grundsätzlich nicht direkt auf den Speicher eines anderen Prozesses zugreifen
Thread ein Ausführungsstrang innerhalb eines Prozesses; ein Prozess kann mehrere Threads besitzen, die sich den Speicher und die Ressourcen des Prozesses teilen — Threads zu wechseln ist deutlich schneller als ganze Prozesse zu wechseln
Task allgemeiner Begriff für eine auszuführende Arbeitseinheit — wird in der Praxis oft uneinheitlich synonym zu „Prozess“ verwendet (z. B. Task-Manager, Taskplaner)

Multitasking

Multitasking bedeutet, dass mehrere Prozesse scheinbar gleichzeitig ausgeführt werden. Auf einem Ein-Kern- Prozessor geschieht das durch sehr schnelles Umschalten zwischen Prozessen (Kontextwechsel); auf Mehrkern-Prozessoren können zusätzlich mehrere Prozesse tatsächlich parallel auf unterschiedlichen Kernen laufen.

Kooperatives vs. präemptives Multitasking

ModellFunktionsweiseRisiko
Kooperatives Multitasking (älteres Modell, z. B. frühe Windows-Versionen) jeder Prozess gibt die CPU freiwillig ab, sobald er wartet oder „fertig“ ist ein fehlerhafter oder hängender Prozess gibt die CPU nie ab und blockiert dadurch das gesamte System
Präemptives Multitasking (moderner Standard, z. B. aktuelles Windows/Linux/macOS) der Scheduler des Betriebssystems entzieht jedem Prozess nach einer festen Zeitscheibe zwangsweise die CPU, unabhängig vom Prozess selbst deutlich stabiler — ein hängender Prozess blockiert in der Regel nur sich selbst, nicht das gesamte System

Praxisbeispiel

Ein Nutzer meldet dem Support der TechNord GmbH, eine ältere Anwendung friere ab und zu ein — er befürchtet, dass dadurch der ganze PC lahmgelegt wird:

  • Erklärung durch den Support: Da das aktuelle Betriebssystem präemptives Multitasking nutzt, sollte die hängende Anwendung isoliert bleiben — im Task-Manager lässt sie sich gezielt beenden, ohne andere Programme zu beeinträchtigen.
  • Falls tatsächlich das gesamte System einfriert, deutet das auf ein tieferliegendes Problem hin, z. B. einen fehlerhaften Treiber, der im privilegierten Kernel-Modus läuft und dadurch nicht durch normales Prozess-Scheduling isoliert werden kann.
  • Moderne Webbrowser nutzen zudem für jeden Tab einen eigenen Prozess bzw. Thread — stürzt ein einzelner Tab ab, bleibt der Rest des Browsers unbeeinträchtigt nutzbar.

Merksatz

Ein Prozess hat eigenen isolierten Speicher, ein Thread läuft innerhalb eines Prozesses und teilt sich dessen Ressourcen — Thread-Wechsel sind günstiger als Prozess-Wechsel. Kooperatives Multitasking verlässt sich darauf, dass Prozesse die CPU freiwillig abgeben (Risiko: Systemblockade). Präemptives Multitasking entzieht zwangsweise per Scheduler nach fester Zeitscheibe — der moderne, stabilere Standard.
