Zeichensätze — ASCII, Unicode, UTF-8 Kern AP1

Einstieg

Ein Byte ist nur eine Zahl von 0 bis 255 — kein Naturgesetz legt fest, dass die Zahl 65 den Buchstaben „A“ bedeutet. Das ist reine Vereinbarung, wie ein gemeinsames Geheimcode-Buch zwischen zwei Personen. ASCII war das erste kleine Codebuch, das nur englische Zeichen kannte. Unicode ist das große, weltweite Codebuch für praktisch jede Schrift der Erde. Und UTF-8 ist die platzsparende Art, dieses riesige Codebuch tatsächlich in Bytes aufzuschreiben — kurze Codes für häufige Zeichen, längere für seltene.

Konzept

Das Grundproblem

Ein Byte kann 256 verschiedene Werte annehmen (0–255). Welcher Wert welchem Buchstaben entspricht, ist keine physikalische Vorgabe, sondern eine Vereinbarung — Kodierungsstandards legen diese Zuordnung fest.

ASCII (1963)

ASCII definiert 128 Zeichen mit den Werten 0–127 (7 Bit):

BereichInhalt
0–31Steuerzeichen, nicht druckbar (z. B. 0x09 = Tab, 0x0A = LF, 0x0D = CR)
32–64Leerzeichen, Sonderzeichen, Ziffern 0–9 (48–57)
65–90Großbuchstaben A–Z (A = 65 = 0x41)
97–122Kleinbuchstaben a–z (genau 32 höher als Großbuchstaben)

Wichtige Falle: Das Zeichen „0“ hat den ASCII-Wert 48 (0x30) — die Zahl 0 hat den Wert 0. Das sind zwei völlig verschiedene Dinge, ein häufiger Programmierfehler.

ASCII deckt nur englische Zeichen ab — kein Ä, Ö, Ü, ß, keine chinesischen oder arabischen Schriftzeichen. Jahrzehntelang entwickelte jedes Land eigene 256-Zeichen-Erweiterungen (z. B. ISO-8859-1, Windows-1252), was zu erheblichen Kompatibilitätsproblemen führte.

Unicode (1991)

Unicode definiert einen einheitlichen Zeichensatz für praktisch alle Schriftsysteme der Welt (über 140.000 Zeichen). Jedes Zeichen hat einen eindeutigen Codepoint, geschrieben als U+XXXX — z. B. A = U+0041, ä = U+00E4, € = U+20AC.

UTF-8 — die effiziente Kodierung

Unicode definiert nur, welche Zeichen es gibt — nicht, wie sie als Bytes gespeichert werden. UTF-8 ist die mit Abstand verbreitetste Kodierung (über 98 % aller Webseiten) und arbeitet variabel lang: einfache Zeichen brauchen 1 Byte, seltenere bis zu 4 Bytes.

ZeichenCodepointUTF-8-BytesAnzahl Bytes
AU+0041411
äU+00E4C3 A42
€U+20ACE2 82 AC3
😀U+1F600F0 9F 98 804

Alle ASCII-Zeichen (0–127) werden in UTF-8 mit genau einem Byte und identischem Bitmuster dargestellt — jede reine ASCII-Datei ist deshalb automatisch gültiges UTF-8 (bewusstes Design, keine zufällige Kompatibilität). Zur Abgrenzung: UTF-16 nutzt mindestens 2 Bytes pro Zeichen (intern u. a. bei Windows/Java), UTF-32 immer 4 Bytes — UTF-8 ist für westliche Texte am effizientesten und deshalb Web-Standard.

CR, LF und CRLF: die Sache mit dem Zeilenumbruch

CR und LF stammen aus der Ära mechanischer Schreibmaschinen und Fernschreiber: CR (Carriage Return, 0x0D) bewegte den Schreibwagen zurück an den linken Rand, LF (Line Feed, 0x0A) schob das Papier eine Zeile weiter — für einen vollständigen Zeilenumbruch brauchte man beide Aktionen nacheinander.

Windows hat CRLF von MS-DOS übernommen, das wiederum von CP/M (ausgelegt für Fernschreiber) geprägt wurde. Linux/Unix wählten das effizientere LF allein. Diese Entscheidung von ca. 1980 verursacht bis heute Probleme: Ein unter Windows bearbeitetes Shell-Skript zeigt unter Linux oft ein sichtbares ^M am Zeilenende (das übrig gebliebene CR), und Git kann bei reiner Zeilenend-Konvertierung riesige, irreführende Diffs anzeigen.

Praxisbeispiel

Ein Entwickler der TechNord GmbH erhält von einem Windows-Kollegen ein Shell-Skript, das unter Linux mit einer kryptischen Fehlermeldung abbricht:

  • Diagnose: cat -A skript.sh zeigt am Ende jeder Zeile ^M$ statt nur $ — das Skript enthält Windows-Zeilenenden (CRLF) statt der unter Linux erwarteten LF.
  • Ursache: Der Linux-Interpreter interpretiert das übrig gebliebene CR (0x0D) fälschlich als Teil des Befehls.
  • Lösung: Mit sed oder dem Werkzeug dos2unix werden alle CRLF in reines LF umgewandelt, danach läuft das Skript fehlerfrei.

Ein weiterer Fall: Eine Exportdatei aus einem älteren System zeigt beim Öffnen falsche Umlaute (Ä wird zu einem kryptischen Symbol). Der Grund: Die Datei wurde in ISO-8859-1 statt UTF-8 exportiert — beim Import muss die korrekte Kodierung explizit angegeben werden, damit die Bytes richtig interpretiert werden.

Merksatz

ASCII: 128 Zeichen, nur Englisch, 7 Bit. Unicode: Codepoints (U+XXXX) für praktisch alle Schriftsysteme. UTF-8: variable 1–4-Byte-Kodierung von Unicode, rückwärtskompatibel mit ASCII, Web-Standard. Linux/macOS nutzen LF, Windows CRLF, altes macOS CR — falsche Zeilenenden erzeugen das ^M-Problem in Shell-Skripten und unnötige Git-Diffs.
