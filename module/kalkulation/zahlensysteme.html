Zahlensysteme Kern AP1 AP2

Einstieg

Eine Uhrzeit lässt sich auf ganz unterschiedliche Arten darstellen: als Ziffern auf einem digitalen Display (12:45), als Zeigerstellung auf einem analogen Ziffernblatt, oder als Bitfolge auf einer LED-Digitaluhr, die nur leuchtende und dunkle Segmente kennt. In allen drei Fällen ist exakt dieselbe Uhrzeit gemeint — nur die Darstellung unterscheidet sich. Genauso verhält es sich mit Zahlensystemen: Dieselbe Zahl lässt sich dezimal, binär oder hexadezimal aufschreiben — der Wert bleibt identisch, nur die Symbole und die Basis des Zählsystems ändern sich.

Binärzahlen lesen

Jede Stelle einer Binärzahl hat einen Stellenwert, der sich von rechts nach links verdoppelt. Man addiert die Stellenwerte aller Stellen, an denen eine 1 steht:

Stellenwert1286432168421
Bit10110010

0b10110010 = 128 + 32 + 16 + 2 = 178 (Dezimal).

Hexadezimalzahlen lesen

Jede Hex-Stelle entspricht genau 4 Bit (einem Nibble). Deshalb lässt sich ein Byte immer mit genau 2 Hex-Stellen darstellen — das macht Hex so praktisch für die kompakte Darstellung von Speicherinhalten.

0xB2 = B · 16 + 2 = 11 · 16 + 2 = 178 (Dezimal) — derselbe Wert wie oben, nur hexadezimal notiert.

Zweierkomplement: negative Zahlen binär darstellen

Ein reiner Binärcode kennt zunächst kein Vorzeichen. Um auch negative Zahlen darzustellen, verwenden Computer das Zweierkomplement. Bei einer 8-Bit-Zahl gilt das höchstwertige Bit (ganz links) als Vorzeichenbit: 0 = positiv, 1 = negativ. Damit lassen sich mit 8 Bit Werte von −128 bis +127 darstellen.

Bildung des Zweierkomplements einer negativen Zahl:

  1. Betrag der Zahl binär darstellen (z. B. 5 = 00000101).
  2. Alle Bits invertieren (0↔1): 00000101 → 11111010 (Einerkomplement).
  3. 1 addieren: 11111010 + 1 = 11111011.

11111011 stellt im Zweierkomplement also −5 dar. Der Vorteil: Der Prozessor kann Addition und Subtraktion mit derselben Rechenschaltung ausführen, ohne Vorzeichen gesondert behandeln zu müssen — (+3) + (−5) wird einfach binär addiert und ergibt korrekt −2.

Praxisbeispiel

Ein IT-Support-Mitarbeiter erhält eine Datei namens rechnung.pdf, die sich jedoch nicht öffnen lässt. Der Verdacht: Jemand hat die Dateiendung manuell geändert, ohne dass der Inhalt wirklich ein PDF ist.

Jedes Dateiformat beginnt mit einer festen Byte-Folge, den sogenannten Magic Bytes (Dateisignatur) — sie verraten den echten Dateityp unabhängig von der Endung. Ein PDF beginnt beispielsweise immer mit den Hex-Bytes 25 50 44 46 (ASCII „%PDF“). Mit einem Hex-Editor oder dem Kommandozeilenbefehl file lassen sich die ersten Bytes einer Datei prüfen — stimmen sie nicht mit den erwarteten Magic Bytes überein, war die Umbenennung tatsächlich nur kosmetisch, und der reale Dateityp ist ein anderer.

Typische Prüfungsfrage: „Warum reicht das Umbenennen der Dateiendung nicht aus, um den tatsächlichen Dateityp zu ändern?“ – Antwort: Die Dateiendung ist nur ein Hinweis für das Betriebssystem, welches Programm die Datei öffnen soll. Der echte Dateityp steckt in den ersten Bytes der Datei selbst (Magic Bytes) und bleibt beim Umbenennen unverändert.

Merksatz

Dezimal (Basis 10), Binär (Basis 2), Hexadezimal (Basis 16) beschreiben denselben Wert mit unterschiedlichen Symbolen. Ein Byte = genau 2 Hex-Stellen. Das Zweierkomplement stellt negative Zahlen dar: Bits invertieren, dann 1 addieren — mit 8 Bit sind so Werte von −128 bis +127 darstellbar.
