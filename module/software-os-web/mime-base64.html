MIME-Types & Base64 AP1

Einstieg

Ein MIME-Type ist wie das Etikett auf einem Paket, das dem Zusteller sagt, wie er es behandeln soll — zerbrechlich, kühl lagern, sofort öffnen. Ohne dieses Etikett wüsste ein Browser nicht, ob er eine empfangene Datei anzeigen, ausführen oder zum Download anbieten soll. Base64 wiederum ist wie das Übersetzen einer Nachricht in ein einfaches Alphabet aus Buchstaben und Zahlen, damit sie sicher durch Kanäle transportiert werden kann, die eigentlich nur für reinen Text gebaut wurden — kein Schutz, keine Komprimierung, nur ein anderes Alphabet für dieselbe Nachricht.

Konzept

MIME-Types

MIME (Multipurpose Internet Mail Extensions) ist ein Standard, der den Inhaltstyp einer Datei für ein Übertragungsprotokoll beschreibt — ursprünglich für E-Mail entwickelt, heute Standard in HTTP und überall im Web. MIME-Types haben immer das Format typ/subtyp.

MIME-TypeDateiendungBeschreibung
text/html.htmlHTML-Webseite
text/plain.txtreiner Text
text/css.cssCSS-Stylesheet
application/javascript.jsJavaScript
application/json.jsonJSON-Daten
application/pdf.pdfPDF-Dokument
image/jpeg.jpg, .jpegJPEG-Bild
image/svg+xml.svgSVG-Vektorgrafik
multipart/form-data–Datei-Upload in HTML-Formularen
application/octet-streambeliebigunbekannte Binärdatei — Browser bietet Download an

Übertragen wird der MIME-Type im HTTP-Header Content-Type, z. B. Content-Type: image/jpeg.

Sicherheitsrelevanz: Ein Angreifer könnte eine .exe-Datei fälschlich als image/jpeg deklarieren. Server sollten den MIME-Type deshalb nicht blind vom Client übernehmen, sondern den tatsächlichen Inhalt prüfen (siehe Einheit Zahlensysteme, Magic Bytes). Browser sollten umgekehrt den vom Server angegebenen MIME-Type respektieren (Header X-Content-Type-Options: nosniff).

Base64

Base64 wandelt Binärdaten in eine Zeichenkette aus 64 druckbaren ASCII-Zeichen um (A–Z, a–z, 0–9, +, /) — es wird dabei nichts komprimiert oder verschlüsselt, nur umgeformt. Grund: Manche Übertragungsprotokolle wurden ursprünglich nur für reinen Text entworfen, in denen binäre Steuerzeichen die Übertragung stören könnten.

EinsatzBeispiel
E-Mail (SMTP)binäre Anhänge werden als Base64 kodiert
Web/HTTPData-URLs betten kleine Bilder direkt in HTML/CSS ein (data:image/png;base64,...)
APIs/JSONJSON kann keine Binärdaten enthalten — Bilder/Dateien werden als Base64-String verpackt
TokensJWT (JSON Web Token) ist Base64-kodiertes JSON

Technisch nimmt Base64 je 3 Bytes (24 Bit) und teilt sie in vier 6-Bit-Gruppen auf — jede Gruppe wird einem der 64 druckbaren Zeichen zugeordnet. Dadurch wächst die Datenmenge um etwa 33 % (aus 3 Bytes werden 4 Zeichen): Eine 1 MB große Datei wird als Base64 rund 1,33 MB groß.

Wichtig: Base64 ist keine Verschlüsselung. Jeder kann eine Base64-Zeichenkette in Sekunden zurückwandeln. Ein Base64-kodiertes Passwort ist genauso unsicher wie Klartext — für echte Sicherheit braucht es Verschlüsselung (AES, RSA) oder Hashing (siehe Modul IT-Sicherheit).

Praxisbeispiel

Zwei Situationen bei der TechNord GmbH:

  • Datei-Upload-Formular: Ein Nutzer lädt ein Profilbild hoch, der Browser sendet dabei Content-Type: image/jpeg. Der Server verlässt sich jedoch nicht blind auf diese Angabe, sondern prüft zusätzlich die Magic Bytes der Datei — so wird verhindert, dass eine als Bild getarnte ausführbare Datei unbemerkt hochgeladen wird.
  • Firmenlogo im CSS: Ein Entwickler bettet ein kleines Firmenlogo direkt als Base64-Data-URL ins CSS ein, statt eine separate Bilddatei zu laden — das spart einen zusätzlichen HTTP-Request, macht die CSS-Datei aber rund 33 % größer als das reine Bild.

Merksatz

MIME-Type (typ/subtyp) sagt dem Browser, wie er empfangene Daten behandeln soll — Server sollten ihn nicht blind vom Client übernehmen, sondern den echten Inhalt prüfen. Base64 wandelt Binärdaten in druckbare ASCII-Zeichen um (3 Bytes → 4 Zeichen, ca. 33 % größer) — reine Umformung, keine Verschlüsselung und keine Komprimierung.
