Angriffsmethoden Kern AP1 AP2

Einstieg

Ein Einbrecher hat mehrere Möglichkeiten, in ein Haus zu kommen: Er kann die Tür aufbrechen (Technik angreifen), sich als Handwerker ausgeben und einfach hereingebeten werden (den Menschen täuschen), oder so lange am Klingelbrett drücken, bis jemand entnervt öffnet (das System überlasten). Genau diese drei Grundmuster – Technik angreifen, Menschen täuschen, System überlasten – finden sich auch bei Angriffen auf IT-Systeme wieder.

Konzept

Nicht jeder Angriff nutzt Schadsoftware. Viele zielen direkt auf den Menschen oder auf Schwächen in Anwendungen und Netzwerken:

Bei Social Engineering lohnt sich die feinere Unterscheidung: Phishing (Massenmail), Spear-Phishing (gezielt auf eine Person zugeschnitten), Whaling (Ziel ist die Führungsebene), Pretexting (erfundener Vorwand, z. B. angeblicher IT-Support) und Baiting (präparierter USB-Stick, der „zufällig“ herumliegt). Alle nutzen den Menschen als Schwachstelle, nicht die Technik.

SQL-Injection im Detail

SQL-Injection steht seit Jahren auf Platz 1 der OWASP Top 10 der Web-Sicherheitslücken und entsteht, wenn Benutzereingaben ungefiltert in einen SQL-Befehl eingebaut werden: 

// Unsicher: Eingabe direkt in SQL eingebaut
$sql = "SELECT * FROM users WHERE name='" . $_POST['name'] . "'";

// Angreifer gibt ein: ' OR '1'='1
// Ergebnis: SELECT * FROM users WHERE name='' OR '1'='1'
// '1'='1' ist immer wahr -> gibt ALLE Benutzer zurück!

Die wichtigste Gegenmaßnahme sind Prepared Statements (vorbereitete Anweisungen): SQL-Struktur und Daten werden getrennt übergeben, sodass Benutzereingaben niemals als SQL-Befehl interpretiert werden können. 

// Sicher mit Prepared Statement (PDO)
$stmt = $pdo->prepare("SELECT * FROM users WHERE name = ?");
$stmt->execute([$_POST['name']]);
// ' OR '1'='1 wird nur als Suchtext behandelt, nicht als SQL-Befehl

Weitere Best Practices gegen SQL-Injection und ähnliche Angriffe: minimale Rechte pro Datenbanknutzer (Least Privilege), regelmäßige Backups nach der 3-2-1-Regel, keine Datenbankfehlermeldungen nach außen zeigen (Angreifer lernen sonst die Tabellenstruktur) und Eingaben zusätzlich validieren (eine E-Mail-Adresse muss wie eine aussehen, eine ID muss eine Zahl sein).

Praxisbeispiel

Freitagnachmittag im IT-Support der „Rheinfeld Elektrotechnik GmbH“: Eine Mitarbeiterin aus der Buchhaltung ruft an.

  • Schritt 1: Sie hat eine E-Mail erhalten, angeblich vom Geschäftsführer, mit der dringenden Bitte, eine Rechnung über 8.500 € an ein „neues“ Konto zu überweisen. Die Absenderadresse sieht auf den ersten Blick korrekt aus. → Richtige Reaktion: Nicht sofort überweisen, sondern die Anfrage über einen zweiten Kanal (Telefonanruf beim Geschäftsführer) verifizieren. Das ist ein klassischer CEO-Fraud, eine Form von Spear-Phishing kombiniert mit Absender-Spoofing.
  • Schritt 2: Kurz darauf lässt sich die Buchhaltungssoftware nicht mehr öffnen, eine Meldung fordert Lösegeld in Kryptowährung. → Das ist Ransomware; sofort den Rechner vom Netz trennen (nicht ausschalten, um Spuren zu sichern) und die IT-Notfallkette informieren, statt selbst zu zahlen.
  • Schritt 3: Eine Analyse zeigt: Der Angreifer kam über eine Kontaktformular-Eingabe auf der Firmenwebsite herein, die ungefiltert in eine SQL-Abfrage eingebaut wurde. → Klassische SQL-Injection; die Website hatte keine Prepared Statements verwendet.

Typische Prüfungsfrage: „Nennen Sie den Angriffstyp und eine passende Gegenmaßnahme.“ Antwort z. B.: SQL-Injection → Prepared Statements; Phishing/CEO-Fraud → Schulung, Verifikation über zweiten Kanal, Spam-Filter, MFA.

Merksatz

Angriffe zielen auf Technik (SQL-Injection, XSS), auf Menschen (Phishing-Varianten) oder auf Verfügbarkeit (DoS/DDoS). Zu jedem Angriff gehört eine bekannte Gegenmaßnahme: Prepared Statements gegen SQL-Injection, Verifikation über zweiten Kanal gegen Phishing/CEO-Fraud.
