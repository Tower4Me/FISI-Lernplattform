UML-Diagramme Kern AP1

Einstieg

UML ist wie ein genormter Satz an Bauplan-Symbolen, den jede Baufirma versteht, egal wer den Plan gezeichnet hat. Ein Use-Case-Diagramm ist wie ein Grundriss, der zeigt, wer durch welche Tür kommt und was dort erledigt wird (Akteure und ihre Interaktionen). Ein Klassendiagramm ist der Bauplan der Rauminnenstruktur — welche Räume gibt es, wie hängen sie zusammen. Und ein Aktivitätsdiagramm ist der Ablaufplan der tatsächlichen Arbeitsschritte im Gebäude — was passiert zuerst, wo wird entschieden, was läuft parallel.

Konzept

UML (Unified Modeling Language) ist eine genormte, programmiersprachenunabhängige grafische Notation zur Modellierung von Software-Systemen. Sie hilft Entwicklern, Kunden und weiteren Beteiligten, ein gemeinsames Verständnis eines Systems zu entwickeln, bevor programmiert wird.

Use-Case-Diagramm

Zeigt, welche Akteure (Nutzer, externe Systeme) mit dem System interagieren und welche Anwendungsfälle (Use Cases) dabei genutzt werden.

ElementDarstellungBedeutung
AkteurStrichmännchenNutzer oder externes System, das mit dem System interagiert
Use CaseEllipseein konkreter Anwendungsfall/eine Funktion des Systems
SystemgrenzeRechteckgrenzt das modellierte System vom Rest der Welt ab

Klassendiagramm

Zeigt die statische Struktur eines Systems: Klassen mit ihren Attributen und Methoden sowie Beziehungen zwischen den Klassen. Eine Klasse wird als Rechteck mit drei Bereichen dargestellt: Klassenname, Attribute, Methoden.

BeziehungstypBedeutung
Assoziationallgemeine Verbindung zwischen zwei Klassen
Generalisierung (Vererbung)eine Klasse erbt von einer allgemeineren Klasse (siehe Einheit Programmierparadigmen)
Aggregation„hat-ein“-Beziehung, Teile können auch unabhängig vom Ganzen existieren
Kompositionstarke „hat-ein“-Beziehung, Teile existieren nicht ohne das Ganze

Aktivitätsdiagramm

Zeigt den Ablauf eines Prozesses oder Algorithmus — ähnlich einem klassischen Flussdiagramm, aber als Teil des UML-Standards. Wichtige Elemente: Startknoten, Aktivitäten (abgerundete Rechtecke), Entscheidungsknoten (Raute — entspricht der Verzweigung aus der Einheit Datentypen & Kontrollstrukturen), parallele Abläufe (Balken für Fork/Join) und Endknoten.

Praxisbeispiel

Die TechNord GmbH plant eine neue Online-Bestellfunktion und erstellt vor der Programmierung drei UML-Diagramme:

  • Use-Case-Diagramm: Der Akteur „Kunde“ nutzt den Use Case „Bestellung aufgeben“, der wiederum per <<include>> zwingend den Use Case „Zahlung prüfen“ einbindet.
  • Klassendiagramm: Die Klassen „Kunde“, „Bestellung“ und „Position“ werden mit ihren jeweiligen Attributen modelliert — eine Bestellung besteht aus mehreren Positionen (Aggregation).
  • Aktivitätsdiagramm für den Bestellprozess: Start → Warenkorb prüfen → Entscheidung „Lagerbestand ausreichend?“ → bei Ja: Zahlung abwickeln, bei Nein: Kunde benachrichtigen → Ende.

Alle drei Diagramme entstehen, bevor eine Zeile Code geschrieben wird — sie dienen der gemeinsamen Abstimmung zwischen Entwicklungsteam und Fachabteilung.

Merksatz

UML ist eine genormte, sprachunabhängige Modellierungsnotation. Use-Case-Diagramm = wer (Akteur) nutzt was (Use Case). Klassendiagramm = statische Struktur mit Klassen, Attributen, Methoden und Beziehungen (Assoziation, Generalisierung, Aggregation, Komposition). Aktivitätsdiagramm = Ablauf mit Start/Ende, Aktivitäten, Entscheidungsknoten und parallelen Abläufen.
