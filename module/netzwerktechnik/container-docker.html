Container (Docker) vs. VM Kern AP1 AP2

Einstieg

Beim Mehrfamilienhaus aus der letzten Einheit hatte jede Wohnung (virtuelle Maschine) ihre eigene komplette Infrastruktur — eigene Küche, eigenes Bad, eigene Heizung. Stell dir jetzt stattdessen ein modernes Großraumbüro mit fest zugewiesenen, aber leichten Trennwänden vor: Jede Firma im Büro hat ihren eigenen abgetrennten Bereich mit eigenem Mobiliar, nutzt aber dieselbe zentrale Heizung, dieselbe zentrale Stromversorgung und denselben zentralen Aufzug wie alle anderen. Das spart Platz und Aufwand, weil nicht jede Firma ihre eigene komplette Gebäudetechnik mitbringen muss. Genau diesen leichteren Ansatz verfolgt Containerisierung im Vergleich zur klassischen virtuellen Maschine.

Konzept

Sowohl virtuelle Maschinen (VMs) als auch Container isolieren Anwendungen voneinander, unterscheiden sich aber grundlegend darin, was genau isoliert wird:

Docker ist die bekannteste Plattform zur Erstellung und Verwaltung von Containern. Eine Anwendung wird dabei mit allen benötigten Bibliotheken und Abhängigkeiten in einem Image gebündelt — dieses Image läuft dann auf jedem System mit Docker-Unterstützung identisch, unabhängig von der zugrunde liegenden Systemumgebung. Das löst das klassische Problem „Bei mir läuft es doch!“, da die Laufzeitumgebung fest im Image enthalten ist.

Wann VM, wann Container?

  • VM — wenn vollständige Isolation nötig ist (z. B. unterschiedliche Betriebssysteme auf derselben Hardware, hohe Sicherheitsanforderungen zwischen Mandanten).
  • Container — wenn Anwendungen schnell, ressourcenschonend und portabel bereitgestellt werden sollen, insbesondere bei vielen gleichartigen, kurzlebigen Instanzen (z. B. Microservices).

In der Praxis werden beide Technologien häufig kombiniert: Container laufen oft innerhalb einer virtuellen Maschine, um die Vorteile beider Ansätze zu nutzen — starke Isolation der VM nach außen, leichte und schnelle Bereitstellung der Container darin.

Praxisbeispiel

Ein Softwareunternehmen entwickelt eine Webanwendung, die aus mehreren kleinen Diensten besteht (Microservices: Login-Dienst, Bestellsystem, Zahlungsanbindung).

  • Jeder Microservice wird als eigenes Docker-Image gebaut, das genau die benötigten Bibliotheken enthält — unabhängig davon, welche Softwareversionen auf dem Zielserver sonst installiert sind.
  • Bei stark steigender Nutzerzahl lassen sich zusätzliche Container-Instanzen des Bestellsystems innerhalb von Sekunden starten, um die Last abzufangen — mit einer klassischen VM würde das Hochfahren deutlich länger dauern.
  • Für einen Kunden mit besonders strengen Sicherheitsanforderungen (getrennte Mandanten) wird zusätzlich eine dedizierte virtuelle Maschine eingesetzt, in der die Container dieses Kunden vollständig von anderen Kunden isoliert laufen.

Typische Prüfungsfrage: „Warum starten Container deutlich schneller als virtuelle Maschinen?“ – Antwort: Container teilen sich den Kernel des Host-Betriebssystems und benötigen deshalb keinen eigenen vollständigen Betriebssystem-Boot wie eine VM, die ein komplettes Gast-Betriebssystem mitbringt.

Merksatz

VM virtualisiert komplette Hardware inkl. eigenem Betriebssystem — groß, langsam startend, stark isoliert. Container teilen sich den Host-Kernel, enthalten nur Anwendung plus Abhängigkeiten — klein, schnell startend, ressourcenschonend. Docker ist die bekannteste Container-Plattform.
