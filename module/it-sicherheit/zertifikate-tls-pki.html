Zertifikate, PKI & TLS Kern AP1 AP2

Einstieg

Ein Notar beglaubigt eine Unterschrift: Er bestätigt mit seinem eigenen guten Namen, dass eine bestimmte Unterschrift wirklich zu einer bestimmten Person gehört. Ohne diese Beglaubigung könnte jeder behaupten, eine beliebige Unterschrift zu besitzen. Im Internet übernimmt eine Zertifizierungsstelle genau diese Rolle: Sie bestätigt, dass ein öffentlicher Schlüssel wirklich zu einer bestimmten Website gehört.

Konzept

Asymmetrische Verschlüsselung und digitale Signaturen lösen viele Probleme – aber ein Problem bleibt: Woher weiß man, dass ein öffentlicher Schlüssel wirklich zu „firma.de“ gehört und nicht zu einem Angreifer, der sich als „firma.de“ ausgibt?

Eine Zertifizierungsstelle (Certificate Authority, CA) prüft die Identität eines Antragstellers und bestätigt die Zuordnung „dieser öffentliche Schlüssel gehört zu dieser Domain“ mit einem digital signierten Zertifikat. Der Browser vertraut einer festen Liste bekannter CAs. Diese gesamte Vertrauenskette – von der Root-CA über Zwischenzertifikate bis zum Website-Zertifikat – nennt man PKI (Public Key Infrastructure).

TLS und HTTPS

HTTPS ist HTTP über TLS (Transport Layer Security). Beim Verbindungsaufbau (TLS-Handshake) prüft der Client anhand des Zertifikats, ob der Server echt ist (Authentizität), und beide Seiten handeln einen symmetrischen Sitzungsschlüssel aus – genau die hybride Verschlüsselung, die auch bei symmetrischer/asymmetrischer Kryptographie beschrieben wird.

Das Schloss-Symbol im Browser bedeutet: Die Verbindung ist verschlüsselt und das Zertifikat ist gültig. Es bedeutet nicht automatisch, dass die Website selbst vertrauenswürdig oder seriös ist – auch Phishing-Seiten können ein gültiges Zertifikat besitzen.

BegriffRolle
ZertifikatDigital signierte Bestätigung: „Dieser öffentliche Schlüssel gehört zu dieser Domain“
CA (Certificate Authority)Stelle, die die Identität prüft und das Zertifikat signiert
PKI (Public Key Infrastructure)Gesamte Vertrauenskette aus Root-CA, Zwischen- und Endzertifikaten
TLSProtokoll, das Zertifikatsprüfung + hybride Verschlüsselung für sichere Verbindungen kombiniert

Praxisbeispiel

Ein Kunde ruft den Online-Shop „musterhandel.de“ auf und sieht ein Schloss-Symbol in der Adressleiste.

  • Der Browser hat beim TLS-Handshake das Zertifikat von „musterhandel.de“ geprüft und festgestellt, dass es von einer vertrauenswürdigen CA signiert wurde – der Server ist also authentisch.
  • Client und Server haben dabei einen symmetrischen Sitzungsschlüssel ausgehandelt, mit dem die Bestelldaten anschließend schnell und verschlüsselt übertragen werden.
  • Ein IT-Azubi im Support erklärt einem verunsicherten Kunden: „Das Schloss bedeutet, dass niemand die Übertragung mitlesen kann und der Server nachweislich der ist, für den er sich ausgibt. Ob der Shop selbst seriös ist, sagt das Schloss allein nicht aus.“
  • Läuft das Zertifikat ab oder wurde es von keiner vertrauten CA signiert, zeigt der Browser eine Warnung – ein typisches Zeichen für eine mögliche Phishing-Seite oder ein Konfigurationsproblem.

Merksatz

Ein Zertifikat bestätigt (signiert von einer CA), dass ein öffentlicher Schlüssel zu einer bestimmten Domain gehört. Die gesamte Vertrauenskette heißt PKI. TLS nutzt Zertifikate zur Serverprüfung und danach hybride Verschlüsselung für die Datenübertragung (HTTPS = HTTP über TLS).
