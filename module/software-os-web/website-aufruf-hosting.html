Website-Aufruf & virtuelle Hosts Randwissen AP1

Einstieg

Eine Website aufzurufen ist wie Essen zu bestellen: Du kennst die Adresse des Restaurants nicht auswendig, also schlägst du sie nach (DNS wie ein Telefonbuch). Dann schickst du deine Bestellung an genau diese Adresse (Verbindung + Anfrage). In demselben Gebäude teilen sich aber oft mehrere Restaurants dieselbe Lieferadresse — erst der Zusatz auf deiner Bestellung („für Restaurant XY“) sorgt dafür, dass die richtige Küche liefert. Genau das leisten virtuelle Hosts für mehrere Websites auf einem einzigen Server.

Konzept

Ablauf eines Website-Aufrufs

  1. URL-Eingabe: Der Nutzer gibt z. B. https://www.technord.de in den Browser ein.
  2. DNS-Auflösung: Der Domainname wird über DNS in eine IP-Adresse übersetzt (siehe Modul Netzwerktechnik, DNS-Grundlagen).
  3. Verbindungsaufbau: Der Browser baut eine TCP-Verbindung zur Ziel-IP auf — bei HTTPS zusätzlich einen TLS-Handshake für die verschlüsselte Verbindung (siehe Modul IT-Sicherheit, Zertifikate & TLS/PKI).
  4. HTTP-Request: Der Browser sendet eine Anfrage (z. B. GET) an den Server — inklusive eines Host-Headers, der den angeforderten Domainnamen enthält.
  5. Serververarbeitung & HTTP-Response: Der Server verarbeitet die Anfrage und schickt eine Antwort mit Statuscode und Inhalt (HTML, CSS, JavaScript, Bilder) zurück.
  6. Rendering: Der Browser stellt die empfangene Seite dar und lädt bei Bedarf weitere Ressourcen (Bilder, Skripte) mit zusätzlichen Anfragen nach.

Virtuelle Hosts (Virtual Hosts)

Mehrere Websites können auf demselben physischen Server und derselben IP-Adresse betrieben werden — das nennt man virtuelles Hosting. Beim namensbasierten virtuellen Hosting unterscheidet der Webserver die angeforderte Website anhand des Host-Headers im HTTP-Request: Auch wenn zwei Domains dieselbe IP-Adresse teilen, weiß der Server anhand dieses Headers, welche der auf ihm gehosteten Websites tatsächlich ausgeliefert werden soll.

Praxisbeispiel

Die TechNord GmbH betreibt auf einem einzigen Webserver gleichzeitig die Firmenwebsite (www.technord.de) und ein Kunden-Support-Portal (support.technord.de) — beide teilen sich dieselbe öffentliche IP-Adresse:

  • Fragt ein Browser support.technord.de an, liest der Webserver den Host-Header der Anfrage aus und liefert die richtige Website aus — obwohl technisch dieselbe IP-Adresse angesprochen wurde.
  • Fehlerdiagnose: Ein Kunde meldet, die Website sei nicht erreichbar. Der Support geht den Ablauf Schritt für Schritt durch: Löst der Domainname korrekt per DNS auf? Lässt sich überhaupt eine TCP-Verbindung aufbauen (z. B. mit telnet oder ping testbar)? Welchen HTTP-Statuscode liefert der Server zurück? So lässt sich eingrenzen, ob DNS, Netzwerk oder der Webserver selbst die Ursache ist.

Merksatz

Ablauf eines Website-Aufrufs: DNS-Auflösung → Verbindungsaufbau (TCP, ggf. TLS) → HTTP-Request (mit Host-Header) → HTTP-Response → Rendering. Virtuelle Hosts erlauben mehrere Websites auf derselben IP-Adresse — der Webserver unterscheidet sie anhand des Host-Headers im HTTP-Request.
